Immediate priorities recorded, in order: **Enemy Variety → Combat Feel → Skill Variety → Build Identity → Loot Foundations → Boss Encounter.**

## 2026-07-13 — Player-driven character-start skill allocation (PC-016, DEC-033, DONE — playtest-approved)

- **A new character no longer starts with its Active skills already chosen.** The **Starting Direction screen is now the initial Skill Point allocation step**: the player spends their 2 starting Skill Points (DEC-020) on starting skills and explicitly confirms before the first run. Realises the "build is *chosen*, a starting vector not a class" intent (DEC-021) and closes the starting-skill item PC-015 explicitly deferred.
- **Root cause found & fixed.** Default skills came from `SkillDefinition.StartingLevel = 1` on `FractureBolt.asset` + `DetonationField.asset` (seeded by `SkillProgression.Awake`), **and** Active skills were **ungated by level** (rank-0 still fired at index-0 tuning — the PC-015-noted gap). Fix: both actives moved to `StartingLevel = 0`, and each Active skill now activates only at **rank ≥ 1** (`IsUnlocked` guard in `FractureBolt`/`DetonationField`). A fresh character is basic-weapon-only until they invest.
- **Data-driven starting set (not hardcoded).** New `SkillDefinition.AvailableAtCharacterStart` flag decides which skills the start step offers; toggling assets adds/removes starting skills with no code change. Current set: Fracture Bolt + Detonation Field. The Volatile Fracture passive is excluded and **remains reachable via normal leveling** — no hard unlock lock was added (that would regress the approved passive; left as a future seam).
- **Spending optional, confirm explicit.** Points may be left unspent (valid weapon-only start) but Confirm is required; the run can't begin until `SessionContext.InitialAllocationConfirmed`. Future save/load seam: a loaded character sets that flag to skip the forced step (mirrors `SkillPoints.LoadSavedBalance`).
- **Architecture.** Reuses the existing `SkillProgression` core through the `CombatBridge` seam only — **no second skill-tree system**. The cosmetic "Vanguard" node was replaced by the allocation UI (greybox, reuses the front-end's button pattern). Changed: `Skills/SkillDefinition.cs`, `Skills/FractureBolt.cs`, `Skills/DetonationField.cs`, `FrontEnd/CombatBridge.cs`, `FrontEnd/SessionContext.cs`, `FrontEnd/FrontEndUI.cs`, and the three skill assets.
- **Unity validation (in-editor, 0 compile errors):** fresh character shows FractureBolt/DetonationField at **Lv 0**, both `IsUnlocked = false`, Skill Points **2**; `CharacterStartSkills` returns exactly the 2 flagged skills (passive excluded); allocating via the bridge raises rank + unlocks + spends points, and is refused at 0 points; the StartingDirection screen builds 2 allocation rows and Confirm sets `InitialAllocationConfirmed` + advances to World Map. **On-screen UI feel + full playthrough reserved for Yoav's hands-on playtest.** Decision: [DECISIONS.md](DECISIONS.md) DEC-033.

## 2026-07-13 — Death XP penalty: lose sub-level progress on failure, never a level (PC-016, DEC-032, DONE — playtest-approved)

- **Failure now has a felt XP cost.** On Operation failure by player death, the character loses a **configurable flat fraction of the current level's XP bar** (default **20%**), capped so it **never drops below the current level's floor** — levels, total Skill Points, Skill Tree allocations, and Banked rewards are never lost. Example: Level 10 at 40% → Level 10 at 20%. This **resolves the number DEC-019 left open** and **supersedes the prior "XP fully retained on failure"** behaviour that was live since Sprint 2.7.
- **Design review first (no rushed code).** A full read of the progression stack found the permanent-vs-run separation the long-term ARPG/extraction vision needs is **already built and idiomatic**: permanent = `PlayerXP` (cumulative) → `PlayerLevel` (derived) → `SkillPoints`/`SkillProgression`/`SkillDefinition` (allocations keyed for future save) + `PlayerStats`; run-scoped secure-or-lose = the `OperationController` lifecycle events consumed by `CurrencyWallet` (Unsecured/Banked) and `WeaponModule` (Equipped/Banked). Future Inventory/Stash/Rarity/Affix attach as new run-scoped owners that bank on extraction — the proven `WeaponModule` pattern. **No speculative systems were built** (Game-Director "no overengineering").
- **Architecture.** New run-scoped rule `DeathXPPenalty` subscribes to the existing `OperationController.OperationFailed` and **does not modify the Operation lifecycle** (same pattern as `CurrencyWallet`/`WeaponModule`). `PlayerXP` gains a level-agnostic `RemoveXP`; the "never de-level" cap lives in `DeathXPPenalty`, which reads `PlayerLevel`'s current-level boundaries. `OperationHud` gained one failure-summary line ("XP progress lost: -N") so the penalty is visible, not silent. Single-player scope (matches `EnemyXPReward`/`PlayerReference`); the party-wipe rule is a future change to this one component. New file: `Assets/Systems/Progression/DeathXPPenalty.cs`. Changed: `PlayerXP.cs`, `Utilities/OperationHud.cs`. **No networking selected; no save system implied** (DEC-016/020 boundary intact).
- **New characters start with configurable Skill Points (default 2).** `SkillPoints` gained a `startingPoints` field (default 2) granted **exactly once per component instance** in `Awake`, guarded by an `initialized` flag — never re-granted on `OnEnable`/re-enable, Operation reset, death, or respawn (none create a new instance). Temporary new-character model documented: with no save system yet, a fresh instance == a new character (New Character = scene reload, DEC-031). Future save/load seam added: `LoadSavedBalance(int)` SETS (never adds) the balance and suppresses the starter, so loading an existing character can't duplicate the grant — order-independently. Changed: `Skills/SkillPoints.cs`.
- **Governance flags (Game-Director-directed insert).** This reverses playtest-approved behaviour and lands **off the pre-existing roadmap sequence** ("resume Cluster C") while the **Cluster B Decision Gate is intentionally still open** — the failure penalty is part of the mechanic that gate judges. Proceeded at the Game Director's explicit direction (chosen: implement now + record the decision). Decision: [DECISIONS.md](DECISIONS.md) DEC-032. **Not yet DONE** — requires Yoav's hands-on playtest per the Definition of Done. **Unity wiring + validation pending an editor connection** (MCP found no running Unity instance): the one wiring step is adding `DeathXPPenalty` to `Assets/Prefabs/Player/Player.prefab` (all references auto-resolve; `startingPoints` defaults to 2). The penalty **fraction is prototype tuning, not a balance decision**.

## 2026-07-12 — Design architecture review: pillars, not a single law (UNDER REVIEW, non-canonical)

- A design discussion that began with Sprint 2.11 enemy design abstracted into a question about the *shape* of Point Clear's design philosophy. Conclusion reached: **Point Clear is guided by multiple *scoped Design Pillars held in deliberate tension*, not by a single governing law.** Repeated attempts to compress the design into one law each broke — the tell being that every "deeper" law was more abstract and less actionable (a principle that explains everything predicts nothing). The structural reason: the game's identity lives in the *tension between* principles (e.g. self-permanence ⟷ fear-of-loss, realized as secure-or-lose), and a single law has no internal tension to hold that.
- **This conclusion is UNDER REVIEW and explicitly non-canonical.** Recorded as an architecture review at [Documentation/Vision/DESIGN_ARCHITECTURE_REVIEW_2026-07-12.md](Documentation/Vision/DESIGN_ARCHITECTURE_REVIEW_2026-07-12.md), which also captures the candidate pillars (provisional), the tensions between them, and a practical (non-canon) teaching posture for the Sprint 2.11 enemy ("reveal a boundary through incompleteness, not lethality").
- **No canonical philosophy document was changed** — PROJECT_BIBLE.md, DESIGN_DNA.md, CORE_PHILOSOPHY.md, GAME_PILLARS.md, and DECISIONS.md are all left unchanged, and no `DEC-###` was added. Promotion into canon is a deliberate, separate decision for the Game Director + documentation process. No task, roadmap, or code change; Sprint 2.11 remains in design and the Cluster B Decision Gate remains open.
